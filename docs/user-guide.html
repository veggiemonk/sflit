sflit — split a huge Go file, safely

Move Go declarations between files through the AST — and refuse any operation that could change what the program means.

This guide walks the main use case end to end: one 1208-line file becomes 11 focused files, with the build green after every step.

app.go is split by sflit -move into keys.go, update.go, view.go and stream.go.

Install

A single self-contained binary. Import management (goimports) is compiled in — nothing else to install.

install
$ go install github.com/veggiemonk/sflit@latest
$ sflit -h

The shape of every invocation is the same: pick declarations out of a source, write them into a sink.

anatomy of a move
$ sflit -source big.go -regex '^Filter' -sink filter.go -move
        └── where from   └── what to pick     └── where to    └── delete from
                                              (created if      source after
                                               absent)         writing

Without -move, sflit copies. For splitting one file inside a package, you always want -move — a same-directory copy would duplicate declarations and is rejected outright.

Why split a huge file at all?

File boundaries don't matter to the Go compiler. They matter enormously to everyone — and every agent — editing the code.

🤖Agent-efficient codebases

After a split, an agent reads only the file that matters. Small files make string-matching edits reliable, and the filenames become a map of the package.

🔀Parallel editing without contention

A 5000-line file is a serialization point. After a split, work on disjoint features touches disjoint files — no merge conflicts, no stomped edits.

✅Reviewable pure-move commits

Every move is re-parsed and reprinted through gofmt with imports updated — verifiably behavior-free. The reviewer checks the partition, not the code.

↩️Reversible by construction

A move that turned out wrong is fixed by the same move with source and sink swapped. No history, no special undo — just another move.

1Design the partition first

The hard part is not running sflit — it is deciding which declarations belong together. Read the file and group declarations by what they own, not by kind.

In the worked example, a 1208-line app.go (the heart of a Bubble Tea TUI) had natural seams along lifecycle stages and features. Name each sink after what it owns — after the split, the filenames are the map of the package:

new.goconstruction & defaults
init.gostartup commands
keys.gokey handling
update.gomessage dispatch
view.gorendering
stream.gostreaming events
submit.goprompt submission
modal.gomodal choices
session.gosession switching
test_shims.go*ForTest helpers
app.gocore: App struct + accessors — 141 lines

What stays behind is the core: the central struct itself and its small accessors. Everything else gets a feature-named home.

2Move feature clusters of methods

Most of the work is -receiver plus an anchored alternation: methods of one type that form a feature.

splitting app.go — watch it type
$ sflit -source app.go -receiver App -regex '^(globalKey|toggleThinking|cycleModel)$' -sink keys.go -move
$ sflit -source app.go -receiver App -regex '^(Update|forwardToSubModels)$' -sink update.go -move
$ sflit -source app.go -receiver App -regex '^View$' -sink view.go -move
moved 7 declarations · imports updated · gofmt clean

How the regex selects — anchors keep the match honest:

-receiver App -regex '^(globalKey|toggleThinking|cycleModel)$'
func (a *App) Update(msg tea.Msg) ·
func (a *App) globalKey(k tea.KeyMsg) ✓ matched
func (a *App) forceUpdateLayout() ·
func (a *App) toggleThinking() ✓ matched
func (a *App) View() string ·
func (a *App) cycleModel() ✓ matched
Selector habits that pay off:
  • Anchor everything (^...$). Unanchored patterns match substrings — Update would also pick forceUpdateLayout.
  • Alternation over multiple runs. One explicit ^(a|b|c)$ list per sink documents the partition in your shell history.
  • A regex prefix groups a family. -regex '^handleModalChoice' moves every modal-choice handler in one go.

3Move a type together with its methods

-receiver alone selects the type declaration and all its methods, so a helper type travels as one unit.

types travel whole
$ sflit -source app.go -receiver sendWriter -sink submit.go -move

# Types without methods work the same way — -receiver still selects the type:
$ sflit -source app.go -receiver pendingToolCall -sink stream.go -move
$ sflit -source app.go -receiver Config          -sink new.go    -move

Need only some methods while the type stays put? Combine both flags: -receiver App -regex '^Filter' moves matching methods of App and leaves the type behind.

4Free functions, vars, and consts

-regex matches any top-level declaration by name — funcs, methods (by method name, any receiver), vars, consts, types.

constructors move as one cluster
$ sflit -source app.go \
    -regex '^(New|defaultThemes|defaultUISource|defaultCellRegistry|defaultBridge|resolveModels)$' \
    -sink new.go -move

Grouped var/const/type blocks are narrowed: the matching specs travel, the siblings stay behind. Comments travel with their declarations — doc comments, //go: directives, in-body comments, all of it.

Exception: a const block using iota or implicit expressions is a blocked split — move the whole block or refactor it first. sflit refuses rather than guessing.

5Exploit naming conventions

A suffix convention turns into a one-line sweep.

everything *ForTest, gone in one move
$ sflit -source app.go -regex 'ForTest$' -sink test_shims.go -move

This is one of the few places an unanchored-at-the-start pattern is exactly right: you want every declaration ending in ForTest, whatever it's called.

6Mirror the tests

_test.go files move the same way, so the test layout can mirror the source layout.

test-file parity
$ sflit -source app_test.go -regex '^TestStream' -sink stream_test.go -move
$ sflit -source app_test.go -regex '^TestKeys'   -sink keys_test.go   -move

stream.go gets stream_test.go, keys.go gets keys_test.go, and so on. In the worked example the 1460-line test file dissolved into 11 feature-named test files and was deleted once every declaration had moved.

One guard you will meet here: a sink's package must match the source's. Moving package app_test declarations into a package app file fails fast with a package mismatch — pick a sink with the right package clause.

7When a move turns out wrong

A reversal is the same move with source and sink swapped. sflit keeps no history; the reversal is just another move, with the same guarantees.

regret-driven development
# cycleModel didn't belong in keys.go after all — send it back:
$ sflit -source keys.go -regex '^cycleModel$' -sink app.go -move

🛡️ What sflit refuses to do

The guarantee is "never change what the program means." Anywhere that guarantee can't hold, sflit refuses instead of guessing — copy and move alike.

⛔

init functions are rejected. Moving them may change package initialization order; copying duplicates init so it runs twice.

⛔

Partial splits of iota/implicit const blocks are rejected. Select the whole block, or refactor the constants manually first.

⛔

Partial splits of multi-name specs are rejected unless each name has a corresponding explicit value.

⛔

Generated files, cgo (import "C"), and dot imports are rejected. Build-constrained files can only move into sinks with identical constraints.

⛔

Same-directory copies are rejected. The source would keep the declarations, the package would gain duplicates and stop compiling. Use -move, or copy into a different directory.

⚡ Fan out without coordination

Safe to run N concurrent invocations on the same files with no external coordination — built for agents splitting in parallel.

Each run hashes source and sink at parse and verifies them under a short per-file lock at commit. If another writer — sflit or not — changed a file in between, the run re-runs against the fresh content, up to -retries times (default 5). Add -json for a structured result an agent can parse.
